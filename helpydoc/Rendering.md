## Дополнение: как рендер устроен сейчас

### Главный принцип

Рендер строится в один проход: getter вызывается один раз, после чего все части интерфейса используют уже готовые данные `map[string]interface{}`.

### Точка входа в рендер

Основной метод окна — `RenderAll(chatID int64)`.

Внутри него:

1. Вызывается getter и получается `data`
2. Рендерится текст (`renderText(data)`)
3. Рендерится клавиатура (`renderWidgets(data)`)
4. Определяется медиа (`renderMedia(data)`)
5. Возвращается итоговый Telegram message (`tgbotapi.Chattable`) с клавиатурой и, при наличии, медиа

Это важно: в маленькие функции рендера передаются только данные, а не getter-функция.

### Рендер текста

`renderText(data)` исполняет шаблон окна на переданных данных.

- если шаблона нет — возвращается пустая строка
- если в шаблоне ошибка — возвращается пустая строка

### Рендер виджетов

`renderWidgets(data)` проходит по всем widget, получает их `ButtonRow` и собирает `InlineKeyboardMarkup`.

Виджеты используют только `data`, пришедшие в вызов рендера.

### Рендер медиа

`renderMedia(data)` работает так:

1. Сначала проверяет widget, которые реализуют `MediaProvider`
2. Если виджет вернул медиа — оно имеет приоритет
3. Если нет — используется статическое медиа окна (если выполняется условие показа)

Условие показа статического медиа проверяется по `HasMediaKey` в тех же данных `data`.

### Как собирается итоговый message

`RenderAll` возвращает:

- обычный текстовый message, если медиа нет
- media message (`photo`/`video`/`audio`/`document`) с `Caption`, если медиа есть

Клавиатура прикрепляется в обоих случаях.

### Что изменилось в связке Window ↔ Widget

Раньше виджеты могли хранить getter и дергать его внутри себя. Сейчас это убрано.

Теперь:

- `Window` отвечает за единичный вызов getter
- `Window` передает `data` в widget только как входные данные рендера
- callback-обработчики виджетов получают актуальные данные через `handler.HandlerData.GetterData`

То есть виджеты больше не зависят от getter-функции напрямую.

### Почему так лучше

1. Меньше скрытых источников данных внутри виджетов
2. Предсказуемый рендер: одна точка получения данных на весь проход
3. Проще сопровождать и отлаживать рендеринг
4. Нет лишних вызовов getter в рамках одного рендера

### Итого

Актуальная модель рендера:

1. `state` определяет окно
2. `Window.RenderAll` один раз получает `data` из getter
3. `Window` рендерит текст, кнопки и медиа отдельными методами на одних и тех же `data`
4. Возвращается единый message для отправки пользователю
